## Problem

Multiple agents running `bn close` simultaneously race on `index.yaml`. With parallel agent dispatch (`bn run --jobs 4`), this is a correctness bug.

## Approach

Add advisory file locking around all index read-modify-write operations.

### Files to modify
- `Cargo.toml` — add `fs2` dependency (or use existing `libc` for flock)
- `src/index.rs` — add `LockedIndex` wrapper type that acquires `.beans/index.lock`, auto-releases on drop
- `src/index.rs` — all save/load_or_rebuild paths go through lock

### Design
- Separate lockfile (`.beans/index.lock`) not the yaml itself
- Advisory POSIX flock, blocking with timeout (5s default)
- Hold lock for shortest duration: load → modify → save → release
- `LockedIndex` wraps the lock guard — auto-releases on drop

### Acceptance Criteria
- [ ] `LockedIndex` type exists in src/index.rs
- [ ] Concurrent bn close on different beans doesn't corrupt index
- [ ] Lock timeout produces helpful error
- [ ] .beans/index.lock gitignored
- [ ] All existing index tests still pass
