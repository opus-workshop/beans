## What to implement

Enhance the `bn run --dry-run` output to show critical path, file conflicts, and effective parallelism per wave. Currently it just lists static waves with bean IDs and token counts.

## Files

## Details

### Enhanced `print_plan()` output

Current output:

Wave 1: 3 bean(s)
  1.1  Define types        (10k tokens)
  1.2  Config parser        (8k tokens)
  1.3  Add CLI flags       (12k tokens)

Wave 2: 1 bean(s)

New output:

Wave 1: 3 bean(s), 2 effective concurrency
  1.1  Define types        (10k tokens)  ⚡ critical
  1.2  Config parser        (8k tokens)   ⊘ conflicts: 1.1 (src/main.rs)

  1.4  Integration tests   (15k tokens)  ⚡ critical

### Implementation

#### 1. Compute and display critical path

Use `compute_downstream_weights()` (from bean 118.1) to identify the critical path.

The critical path is the longest chain by weight. To find it:
1. Find the bean with the highest weight (start of critical path)
2. Walk forward: at each step, pick the dependent with the highest weight
3. Display as: `Critical path: A → B → C (N waves, ~Xk tokens)`

Mark beans on the critical path with `⚡ critical` in the wave listing.

#### 2. Detect and display file conflicts

For each wave, check which beans share `paths`. Group them:
1. For each pair of beans in the wave, check if their `paths` overlap
2. If they do, annotate the bean with `⊘ conflicts: <other_id> (<shared_path>)`
3. Compute "effective concurrency" = max beans that can actually run in parallel (accounting for conflicts)

Computing effective concurrency is a graph coloring problem (NP-hard in general), but for our use case a greedy approach is fine:
1. Sort beans by downstream weight (highest first)
2. Greedily assign them to "slots" — a bean can share a slot only if it doesn't conflict with any other bean in that slot
3. Number of slots = effective concurrency

#### 3. Enhanced JSON output

Add optional fields to the `DryRun` stream event or add new events:
- `critical_path`: array of bean IDs on the critical path
- Per-round: `effective_concurrency`, `conflicts` array

Modify `print_plan_json()` and update the `StreamEvent::DryRun` variant if needed (or add new fields to `RoundPlan`).
