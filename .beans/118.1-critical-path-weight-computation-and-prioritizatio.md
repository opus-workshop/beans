## What to implement

Add a `compute_downstream_weights()` function and use it to prioritize bean scheduling in the ready-queue. Beans that block the most downstream work should start first — this reduces total wall time by keeping the critical path moving.

## Files

- `src/commands/run/wave.rs` — Add the weight computation function
- `src/commands/run/ready_queue.rs` — Use weights in bean selection sorting

## Details

### 1. Add `compute_downstream_weights()` to `wave.rs`

```rust
/// Compute downstream weight for each bean.
/// Weight = 1 + sum of weights of all beans that directly depend on this bean.
/// Beans on the critical path will have the highest weights.
pub(super) fn compute_downstream_weights(
    beans: &[SizedBean],
    index: &Index,
) -> HashMap<String, u32>
```

Algorithm:
1. Build a reverse dependency graph: for each bean, find which other beans depend on it (via `dependencies` and `produces`/`requires`)
2. Topological sort (leaves first)
3. Walk in reverse topological order, accumulating: `weight[b] = 1 + sum(weight[x] for x in reverse_deps[b])`

Edge cases:
- Beans with no dependents: weight = 1
- Beans not in the dispatch set: ignored
- Cycles (shouldn't happen but handle gracefully): treat as weight 1

### 2. Use weights in `ready_queue.rs` bean selection

In `run_ready_queue_direct()`, the weights should be computed once upfront (alongside the `wave_map`). Then modify the ready-bean sorting:

Currently:
```rust
ready_beans.sort_by(|a, b| {
    a.priority
        .cmp(&b.priority)
        .then_with(|| natural_cmp(&a.id, &b.id))
});
```

Change to:
```rust
ready_beans.sort_by(|a, b| {
    a.priority
        .cmp(&b.priority)
        .then_with(|| {
            // Higher weight = more downstream work blocked = schedule first
            let wa = weight_map.get(&a.id).copied().unwrap_or(1);
            let wb = weight_map.get(&b.id).copied().unwrap_or(1);
            wb.cmp(&wa) // Reverse: higher weight first
        })
        .then_with(|| natural_cmp(&a.id, &b.id))
});
```

Priority still wins (it's explicit user intent), but among same-priority beans, critical-path beans go first.
